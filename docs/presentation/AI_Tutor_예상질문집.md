## A. 구현 (Implementation)

### A1. "데이터 격리는 정확히 어떻게 구현하셨나요?"

> JWT 토큰 안에 `student_id` 가 박혀있고, 모든 API 엔드포인트가 `Depends(get_current_user)` 의존성을 거치면서 토큰을 검증하고 `student_id` 를 추출합니다.
> 그 `student_id` 가 모든 DB 쿼리의 `WHERE` 절에 자동으로 들어갑니다. 예를 들어 `DELETE FROM notes WHERE id=? AND student_id=?` 같이 — 본인 토큰의 `student_id` 와 일치하는 행만 영향을 받습니다.
> RAG 인덱스도 디스크 레벨에서 `rag_indexes/<student_id>/` 디렉토리로 분리되어 있고, 영상 파일도 파일명에 `student_id` 가 박혀있어서 다른 ID 로는 path 자체가 매칭되지 않습니다.

### A2. "RAG 파이프라인은 어떻게 구성하셨나요?"

> 4단계입니다.
> 강의자료가 업로드되면 Parent 1000자 / Child 300자로 청킹합니다. 큰 단위는 컨텍스트 보존용, 작은 단위는 정밀 검색용입니다.
> Child 청크들을 BGE-m3 로 임베딩해서 FAISS IndexFlatIP 에 저장합니다. 정규화된 벡터의 내적이라 코사인 유사도와 동등합니다.
> 질의가 들어오면 먼저 FAISS 로 k=20 후보를 뽑고, BGE Reranker v2-m3 로 정밀 재순위해서 상위 3개를 선택합니다.
> 마지막으로 그 3개에 대응되는 Parent 청크들을 컨텍스트로 LLM 에 전달합니다.

### A3. "인강 영상은 어떻게 자동으로 생성하시나요?"

> 5단계 파이프라인입니다.
> 강의자료 텍스트가 RAG 컨텍스트로 들어가면, LLM 이 슬라이드별로 JSON 배열을 생성합니다. 각 슬라이드는 제목, 헤드라인, 불릿 포인트, 나레이션 4개 필드를 가집니다.
> 그 다음 Edge-TTS 의 한국어 SunHi 보이스로 슬라이드별 음성을 만듭니다. 단어 단위 타임스탬프도 함께 받아서 자막 burn-in 에 씁니다.
> PIL 로 슬라이드 이미지를 렌더링하고, ffmpeg concat demuxer 로 오디오와 비디오를 합칩니다. mov_text 자막을 입혀서 최종 mp4 가 완성됩니다.

### A4. "학생 수준 분류는 어떻게 동작합니까?"

> 두 단계입니다.
> 학생이 채팅을 보낼 때마다 LLM 이 그 발화를 5점 척도로 점수화합니다. "전문 용어를 정확히 사용했는가, 깊이 있는 질문인가" 같은 기준입니다.
> 그 점수가 누적되면 RandomForest 분류기가 입문, 중급, 심화 세 단계 중 하나로 분류합니다.
> 분류 결과에 따라 시스템 프롬프트가 달라집니다. 입문이면 "비유와 쉬운 예시로 설명하라", 심화면 "원리와 메커니즘까지 다루라" 같이 — 같은 질문에도 답변 스타일이 학생에 맞춰집니다.

### A5. "시험 인터랙티브 카드 UI 는 어떻게 만드셨나요?"

> 프론트엔드 측에서 LLM 응답을 파싱해서 카드로 렌더링합니다.
> 응답이 `[객관식 5문제]`, `[주관식 5문제]` 라벨로 구분되어 있고, 각 문제는 `1. 문제 내용 ① 보기1 ② 보기2 ...` 형식입니다.
> 정규식으로 라벨을 찾고 각 문제 블록을 추출한 다음, 객관식이면 ①②③④⑤ 보기를 라디오 버튼처럼 렌더링합니다.
> LLM 응답 형식이 변동 가능하기 때문에 — 마크다운 강조 `**[객관식]**`, `1)` 형식 보기, 라벨이 없는 응답까지 — 여러 fallback 을 두어 강건하게 파싱합니다.

### A6. "영상 자동 생성은 사용자가 따로 트리거 안 해도 되나요?"

> 네, 자료 업로드 직후 자동으로 시작됩니다.
> `/upload` 가 성공하면 프론트엔드가 `startVideoBackground()` 를 호출해서 `/video/generate` 를 POST 합니다. 영상 탭으로 전환하지는 않고 — 사용자가 분석 결과를 보는 동안 백그라운드로 진행됩니다.
> 2초마다 `/video/status` 를 폴링해서 진행 상황을 받고, 완료되면 토스트 알림으로 "X분 X초 소요" 라고 알려줍니다.

### A7. "환경별로 다른 모델을 쓸 수 있다고 하셨는데 어떻게 구현하셨나요?"

> 두 가지 방법으로 가능합니다.
> 첫째, 환경변수 `OLLAMA_MODEL` 로 기본 모델을 지정합니다. 서버 시작 시 그 모델로 동작합니다.
> 둘째, 런타임에 사이드바 드롭다운으로 전환 가능합니다. `POST /model` 엔드포인트가 Ollama 에 설치된 모델 목록 중에서만 선택을 허용하고, 모듈 레벨 변수를 동적으로 변경합니다. 그 다음 LLM 호출부터 새 모델이 적용됩니다.
> 코드 한 줄도 안 바뀝니다.

### A8. "문서 파싱이 병렬 추출이라고 하셨는데 정확히 어떻게 동작하나요?"

> PDF 를 예로 들면 — 같은 파일을 PyMuPDF, pdfplumber, pypdf 세 라이브러리가 `ThreadPoolExecutor` 로 동시에 실행합니다.
> 각 라이브러리가 추출한 텍스트 길이를 비교해서 가장 많이 추출한 결과를 채택합니다.
> 만약 세 라이브러리 모두 텍스트가 부족하면 — 스캔본 PDF 일 가능성이 큽니다 — Surya OCR 과 PaddleOCR 을 다시 병렬로 돌립니다.
> 라이브러리 하나가 실패해도 다른 게 살려주는 강건한 구조입니다.

### A9. "분석 토큰 스트리밍은 어떻게 구현하셨나요?"

> Server-Sent Events (SSE) 방식입니다.
> Ollama 가 `stream=true` 로 호출되면 토큰 단위로 응답을 흘려보냅니다. 각 토큰을 JSON 으로 감싸서 `data: ...` 형식으로 클라이언트에 전송합니다.
> 클라이언트는 `EventSource` API 로 받으면서 `onmessage` 핸들러에서 토큰을 화면에 추가합니다.
> ChatGPT 처럼 답변이 흘러나오는 효과 — 사용자가 첫 토큰을 1초 안에 보기 때문에 체감 응답 속도가 훨씬 좋습니다.

## B. 발생했던 어려움 (Challenges)

### B1. "개발 중 가장 까다로웠던 문제는 무엇이었습니까?"

> 새 자료를 업로드해도 화면에 옛 영상이 계속 표시되는 문제였습니다. 강력 새로고침과 캐시 삭제로도 해결되지 않았습니다.
> 원인이 세 군데였습니다. 백엔드 path 매칭, 캐시 헤더, 그리고 프론트엔드 blob 메모리 잔재.
> 3중 방어로 막았습니다 — 영상 파일명에 학생 ID 와 강의자료명을 박아서 path 자체를 다르게 만들고, 응답 URL 에 mtime 기반 cache-buster 를 추가하고, video 태그에 blob 대신 URL 을 직접 박았습니다.
> 결과적으로 옛 영상은 디스크에 보관되지만 화면에는 새 영상만 표시되고, 같은 자료를 재업로드하면 옛 영상이 자동으로 재활용되는 깔끔한 동작이 됐습니다.

### B2. "KD 학습을 폐기하셨다고 하셨는데, 그 결정은 어떻게 내리셨나요?"

> 데이터를 990개 생성한 시점에서 평가 결과를 봤습니다.
> 베이스 모델 평균 7.76점 대비 파인튜닝 후 7.97점 — 0.21점 상승했습니다.
> 그런데 새로 망가진 케이스가 13개 있었습니다. 답변이 잘리거나, 무한 반복되거나, 질문을 echo 하는 패턴이었습니다.
> 분석해보니 KorQuAD 단답형 데이터에 과적합되면서 long-form 응답 능력을 잃은 것이었습니다. catastrophic forgetting.
> DPO 같은 후처리로 고칠 수 있는 문제가 아니라 — 학습 방식 자체를 폐기하고, 같은 시간을 RAG 와 시스템 프롬프트 강화에 투자했습니다. ROI 측면에서 압도적으로 더 효과적이었습니다.

### B3. "vLLM 호환성 문제는 어떻게 해결하셨나요?"

> Kubeflow 컨테이너에 sudo 권한이 없어서 gcc 를 설치할 수 없었습니다. vLLM 의 V1 엔진이 Triton JIT 컴파일을 시도하면서 gcc 가 필요하다고 실패했습니다.
> 우회책으로 `VLLM_USE_V1=0` 환경변수로 V0 엔진을 강제하고, `enforce_eager=True` 옵션으로 컴파일 자체를 우회했습니다.
> 결과적으로 데이터 생성 파이프라인이 정상 동작했습니다.

### B4. "라이브러리 충돌은 어떻게 처리하셨나요?"

> 가장 큰 건 bcrypt 5.x 와 passlib 1.7.x 호환성 문제였습니다. 회원가입 시 500 에러가 났습니다.
> 원인은 bcrypt 5.x 에서 72바이트 stub 테스트가 truncate 안 하고 ValueError 를 던지는 변경이었습니다.
> `requirements.txt` 에 `bcrypt<4.1` 로 버전을 고정해서 해결했습니다.
> rhwp-python 의 FreeType 충돌도 있었는데 — 순수 파이썬 라이브러리 (python-hwpx, pyhwp) 로 폴백해서 해결했습니다.

### B5. "Python 버전 호환성 이슈도 있었다고 하셨는데?"

> `str | None` 같은 PEP 604 타입 힌트를 썼는데 Python 3.10 부터 지원되는 문법입니다.
> Conda base 환경이 3.9 였기 때문에 SyntaxError 가 나면서 서버가 부팅 자체를 못 했습니다.
> 타입 힌트를 제거하는 hotfix 로 호환성을 회복했습니다. 런타임 동작은 동일합니다.

### B6. "큰 LLM 모델이 영어 reasoning 을 노출하는 문제는 어떻게 처리하셨나요?"

> qwen3:30b-a3b 같은 MoE 모델에서 `think: false` 옵션을 무시하고 영어 사고 과정을 응답 앞에 그대로 노출하는 문제가 있었습니다.
> 3중 방어로 처리했습니다.
> 첫째, 시스템 프롬프트에 "한국어로만, 사고 과정 노출 금지" 를 명시했습니다.
> 둘째, user 메시지 끝에 `/no_think` — qwen3 시리즈가 인식하는 매직 워드를 추가했습니다.
> 셋째, 클라이언트에서 stream 받을 때 영어 reasoning prefix 를 자동 필터링하는 함수를 추가했습니다.

### B7. "영상 생성에서 moviepy 가 죽는 문제는 어떻게 처리하셨나요?"

> 빈 narration 슬라이드에서 Edge-TTS 가 빈 오디오를 반환했고, moviepy 가 그걸 합치다가 zero-size array 에러로 죽었습니다.
> 세 가지 가드를 박았습니다 — 빈 narration 은 슬라이드 title 로 폴백, ffprobe 로 실제 audio duration 을 측정, 그래도 0초면 1초로 패딩.
> 또한 moviepy 자체가 너무 느려서 (14분 소요) 결국 ffmpeg concat demuxer 로 전환했습니다. 같은 작업이 2분 16초로 단축됐습니다.

### B8. "JSON 파싱 실패 같은 LLM 응답 변동성은 어떻게 처리하셨나요?"

> 다층 fallback 으로 처리했습니다.
> 영상 스크립트 생성 시 LLM 응답이 마크다운 코드 펜스 안에 오거나, trailing comma 가 있거나, `<think>` 태그 안에서 끝나거나, 마지막 객체가 잘려있는 등 — 변형이 많습니다.
> 1차로 정상 `[...]` 배열을 찾고, 실패하면 trailing comma 제거 → 잘린 마지막 객체 제거 → json-repair 라이브러리로 자동 복구 순으로 시도합니다.
> 그래도 실패하면 단일 객체들을 모아 배열로 만들고, 최후에는 전체 텍스트를 json-repair 에 통째로 던집니다.

## C. 기술 선택 (Why this)

### C1. "왜 기본 모델로 8B 를 선택하셨나요? 32B 와의 사용 기준은?"

> 운영 효율과 응답 속도 때문입니다.
> 일반 강의자료에서는 RAG 가 컨텍스트를 정확히 잡아주기 때문에, 8B 만으로도 응답 품질이 충분합니다. 토큰 생성 속도도 8B 가 32B 보다 약 4배 빠르고 — 사용자 체감이 훨씬 좋습니다.
> 32B 는 어려운 자료를 위한 **고급 탐색 모드** 입니다. 사이드바 드롭다운에서 즉시 전환 가능합니다.
> 즉 8B 가 속도와 효율의 기본값, 32B 가 품질 우선의 고급 옵션입니다.

### C2. "BGE-m3 가 아니라 OpenAI 임베딩 쓰면 더 정확하지 않나요?"

> 세 가지 이유로 BGE-m3 를 선택했습니다.
> 첫째, **온프레미스** — 강의자료가 외부로 나가지 않습니다. 학교나 회사 자료는 보안상 중요한 부분입니다.
> 둘째, **비용** — 학생 한 명당 자료를 임베딩할 때마다 API 비용이 들지 않습니다.
> 셋째, **한국어 성능** — BGE-m3 는 다국어 SOTA 중 하나로, 한국어 강의자료에서도 OpenAI 와 차이가 거의 없습니다.

### C3. "왜 RAG 인가요? 파인튜닝과 비교하면?"

> ROI 차이입니다.
> 파인튜닝(KD)을 시도했는데 990개 데이터로 평균 점수 0.21점 상승, 새로 망가진 케이스 13개였습니다.
> 같은 시간을 RAG 와 시스템 프롬프트 분기에 투자하면 학습 0시간으로 동등하거나 더 좋은 결과가 나왔습니다. 도메인 지식은 RAG 에 맡기고, 모델은 스타일과 톤만 담당하는 게 효과적이었습니다.

### C4. "왜 Ollama 인가요? vLLM 이나 TGI 와 비교하면?"

> 배포 단순성 때문입니다.
> vLLM 은 Triton JIT 컴파일러와 gcc 가 필요해서 — sudo 권한 없는 환경에서는 띄울 수 없습니다.
> Ollama 는 user-space 바이너리 하나로 끝납니다. 어떤 환경에서든 동일한 명령으로 실행됩니다.
> 성능은 vLLM 이 조금 더 빠르지만, 운영 단순성이 그 차이를 상쇄합니다.

### C5. "왜 FastAPI 인가요? Flask 나 Django 와 비교하면?"

> SSE 와 async 지원 때문입니다.
> 분석 결과를 토큰 단위로 스트리밍하려면 async 가 필수입니다. FastAPI 는 `StreamingResponse` 와 async/await 가 1급 시민입니다.
> Django 는 무거운 monolithic 프레임워크라 단순 API 서버에는 과합니다. Flask 는 async 가 추가 구성 필요합니다.
> FastAPI 의 자동 OpenAPI 문서 생성도 큰 장점입니다.

### C6. "왜 ffmpeg 인가요? moviepy 와 비교하면?"

> 속도 차이입니다.
> moviepy 로 영상 합치기에 14분 걸리던 것이 ffmpeg concat demuxer 로 2분 16초로 줄었습니다.
> 또한 moviepy 가 빈 audio 슬라이드에서 zero-size array 에러로 죽는 문제가 있어서 — 안정성 측면에서도 ffmpeg 직접 호출이 우위였습니다.
> mov_text soft subtitle 입히기, WMP 호환 baseline profile 도 ffmpeg 가 옵션 한 줄로 가능합니다.

### C7. "왜 SQLite 인가요? PostgreSQL 이 더 적합하지 않나요?"

> 졸업 프로젝트 규모에서는 SQLite 로 충분합니다.
> 동시 사용자 100명 수준까지는 무난합니다.
> 운영 규모가 커지면 — `DB_PATH` 환경변수로 connection string 만 분리되어 있어서 — `student/db.py` 모듈만 새로 짜면 PostgreSQL 로 마이그레이션 가능합니다.
> 지금 단계에서는 단순성을 선택했습니다.

### C8. "왜 Edge-TTS 인가요? OpenAI TTS 와 비교하면?"

> 비용입니다.
> Edge-TTS 는 Microsoft Edge 음성 합성 무료 API 라 비용이 0입니다.
> OpenAI TTS 는 1000자당 약 $0.015 — 학생 한 명당 영상 1개 (8분 분량 약 4000자) 만들면 약 $0.06 입니다. 학생 100명이면 $6, 1000명이면 $60 — 누적되면 무시할 수 없습니다.
> 음질은 Edge-TTS 의 SunHi 보이스가 매우 자연스러워서 — OpenAI 와 큰 차이가 없습니다.

## D. 모델 & 성능

### D1. "전체 시스템의 응답 속도는 어느 정도입니까?"

> 기능별로 다릅니다.
> RAG 검색 — 학생 자료 인덱스에서 1초 이내 응답.
> 채팅 — 첫 토큰은 1초 이내, 전체 응답은 8B 기준 5~10초 / 32B 기준 20~40초.
> 분석 (요약+개념+시험) — 8B 기준 5~7분 / 32B 기준 20~30분.
> 영상 생성 — 8B 기준 약 5분 / 32B 기준 15~25분.

### D2. "8B 와 32B 의 응답 품질 차이가 실제로 큽니까?"

> 일반 자료에서는 RAG 가 컨텍스트를 정확히 잡아주기 때문에 차이가 크지 않습니다.
> 어려운 자료 — 전공 깊이가 있거나 복잡한 추론이 필요한 경우 — 에서는 32B 가 더 정교한 설명을 생성합니다.
> 한국어 자연스러움은 비슷한 수준이고, 32B 가 좀 더 풍부한 어휘를 씁니다.

### D3. "더 큰 모델 (70B 이상) 을 쓸 계획은 없으신가요?"

> 가능합니다. 코드 변경 없이 `OLLAMA_MODEL=llama3.3:70b` 같이 환경변수만 바꾸면 됩니다.
> 다만 운영 환경 GPU VRAM 한계 — 70B 는 양자화해도 40GB+ 필요 — 가 있어서 일반 GPU 서버에서는 어렵습니다.
> A100 80GB 또는 H100 이상이 필요합니다.

### D4. "MoE 모델 (qwen3:30b-a3b 등) 은 시도해보셨나요?"

> 시도했습니다. 활성 파라미터 3B 라서 속도는 8B 수준이지만 품질은 30B 였습니다.
> 다만 `think: false` 옵션을 무시하고 영어 reasoning 을 응답 앞에 노출하는 문제가 있었습니다.
> /no_think 매직 워드와 클라이언트 필터링으로 우회 가능하지만, 모델 자체 한계라 완전히 제거되지는 않았습니다.
> 현재는 안정성 측면에서 일반 모델 (qwen3:8b / 32b) 만 유지하고 있습니다.

### D5. "여러 사용자가 동시에 영상 만들면 어떻게 됩니까?"

> 단일 Ollama 인스턴스가 LLM 요청을 직렬화하기 때문에 동시 5명 정도까지는 큰 지연 없습니다.
> 그 이상이면 대기 시간이 늘어납니다. 영상 생성은 백그라운드 잡이라 사용자 체감은 적지만, 채팅 응답이 느려질 수 있습니다.
> 운영 규모를 키우려면 Kserve InferenceService 로 LLM 을 분리하고 오토스케일링을 적용하는 게 다음 단계입니다.

### D6. "응답 토큰 한도는 어떻게 결정하셨나요?"

> 모델 크기에 따라 자동 분기됩니다.
> 8B 같은 작은 모델은 `num_predict=16384` 까지 — 영상 스크립트 같은 긴 응답을 한 번에 받을 수 있습니다.
> 32B 같은 큰 모델은 `num_predict=2048` 로 줄여서 — 응답 시간 누적을 방지합니다.
> 모델별 토큰 생성 속도가 4배 차이라, 같은 한도를 적용하면 32B 가 사용자 인내심 한계를 넘습니다.

## E. 데이터 & 학습

### E1. "지원하는 자료 형식은 어떤 것들입니까?"

> 7가지 형식입니다. PDF, HWP, HWPX, PPTX, DOCX, TXT, MD.
> 한국 대학 강의자료에서 만날 수 있는 거의 모든 형식을 커버합니다.
> 각 형식별로 여러 라이브러리를 병렬로 실행해서 — 텍스트를 가장 많이 추출한 결과를 채택하는 강건한 구조입니다.

### E2. "스캔본 PDF 처럼 이미지로만 된 자료도 처리 가능합니까?"

> 가능합니다.
> PyMuPDF, pdfplumber, pypdf 가 텍스트를 충분히 추출하지 못하면 — 자동으로 Surya OCR 과 PaddleOCR 을 병렬로 실행합니다.
> 한국어 OCR 성능이 양호한 두 엔진을 동시에 돌려서 텍스트 많은 결과를 채택합니다.
> 다만 OCR 은 무거운 작업이라 — 일반 텍스트 PDF 대비 처리 시간이 5~10배 늘어납니다.

### E3. "강의자료 임베딩은 매번 다시 만들어집니까?"

> 학생별로 디스크에 영속화됩니다.
> `rag_indexes/<student_id>/` 디렉토리에 parents.json, children.json, faiss.index, meta.json 이 저장됩니다.
> 서버가 재시작되어도 — 학생이 다시 같은 자료를 업로드할 필요 없이 — 디스크에서 자동으로 로드됩니다.

### E4. "다른 학생의 강의자료가 검색 결과에 섞일 수 있나요?"

> 절대 불가능합니다.
> 학생별로 FAISS 인덱스 자체가 분리되어 있어서 — 검색 자체가 본인 인덱스 안에서만 일어납니다.
> 디렉토리 단위 격리이고, JWT 토큰의 student_id 가 검색 함수 인자로 자동 전달됩니다.

### E5. "학습 이력은 어디에 저장되고 어떻게 활용됩니까?"

> 학생별 SQLite 테이블에 저장됩니다.
> 채팅 발화, 노트, 일정, 수준 평가 점수 등이 누적됩니다.
> 누적 데이터가 충분해지면 (50개 이상) 학생별 LoRA 학습을 트리거할 수 있는 코드는 있지만 — 자동 트리거는 아직 활성화 안 했습니다. 다음 단계 과제입니다.

## F. 보안 & 운영

### F1. "JWT 토큰이 탈취되면 어떻게 됩니까?"

> 토큰 만료가 24시간으로 짧게 잡혀있습니다. 탈취된 토큰의 영향 범위가 시간적으로 제한됩니다.
> 또한 SECRET_KEY 가 변경되면 모든 기존 토큰이 무효화되기 때문에 — 긴급 시 운영자가 즉시 모든 세션을 끊을 수 있습니다.
> 운영 단계에서는 토큰을 HttpOnly 쿠키로 옮겨서 XSS 노출도 차단할 계획입니다.

### F2. "비밀번호는 어떻게 저장하시나요?"

> bcrypt 해시로 저장됩니다.
> passlib 의 `CryptContext` 를 사용하고, salt 는 자동 생성됩니다.
> 평문 비밀번호는 DB 에 절대 저장되지 않습니다.

### F3. "SQL 인젝션 같은 공격은 어떻게 막으시나요?"

> 모든 쿼리가 파라미터 바인딩 방식입니다 — `?` placeholder 사용.
> sqlite3 라이브러리가 자동으로 escaping 을 처리하기 때문에 — 사용자 입력이 SQL 로 해석되지 않습니다.
> ORM 안 쓰고 raw SQL 만 써도 안전합니다.

### F4. "강의자료가 외부로 유출될 가능성은 없나요?"

> 없습니다.
> 외부 LLM API 를 호출하지 않습니다. 자료 텍스트가 Ollama 와 BGE 모델 추론에만 들어가고, 둘 다 로컬 서버에서 동작합니다.
> 임베딩 벡터도 학생별 디스크에만 저장되고 외부 전송 없습니다.

### F5. "로그에 민감한 정보가 남나요?"

> 채팅 내용이나 강의자료 텍스트는 로그에 남기지 않습니다.
> 에러 로그에는 — 디버깅 목적으로 — 응답 앞부분 200자가 포함될 수 있어서, 운영 환경에서는 로그 마스킹 또는 별도 처리가 필요합니다.

### F6. "다중 사용자 환경에서 성능은 어떻습니까?"

> 단일 Ollama 인스턴스가 LLM 요청을 직렬화하기 때문에, 동시 5명 이상이면 대기 시간이 늘어납니다.
> 영상 생성은 백그라운드 잡이라 사용자 체감은 적습니다.
> 운영 규모를 키우려면 Kserve InferenceService 로 LLM 을 분리하고 오토스케일링을 적용하는 게 다음 단계입니다.

## G. UX & 사용성

### G1. "사용자가 직접 조작해야 하는 부분은 무엇인가요?"

> 회원가입과 로그인, 그리고 강의자료 업로드 — 이 세 가지뿐입니다.
> 자료를 업로드하면 분석과 영상 생성이 모두 자동으로 시작됩니다.
> 채팅이나 시험문제 풀기 같은 학습 인터랙션은 사용자가 원할 때 합니다.

### G2. "모바일에서도 동작합니까?"

> 동작은 합니다 — Tailwind CSS 기반 반응형 SPA 라 모바일 브라우저에서도 열립니다.
> 다만 영상 재생이나 시험 카드 UI 가 데스크톱 화면 기준으로 최적화되어 있어서 — 모바일 전용 레이아웃은 아직 추가하지 못했습니다.

### G3. "처음 사용하는 학생에게 친절합니까?"

> 직관적인 단계별 UI 입니다.
> 자료 업로드 후 "분석 진행 중" 진행 바, "영상 생성 X분 X초 경과" 같은 상태 표시가 명확합니다.
> 시험 문제는 "남은 문제 N개", "채점하기" 버튼 같은 가이드가 있어서 — 별도 튜토리얼 없이도 사용 가능합니다.

### G4. "오류가 발생하면 사용자에게 어떻게 보여줍니까?"

> 토스트 알림과 인라인 메시지를 조합합니다.
> 파싱 실패 같은 부분 오류는 — "분석 다시 실행" 버튼이 있는 호박색 배너로 — 사용자가 직접 재시도할 수 있습니다.
> 서버 오류 같은 치명적 오류는 빨간 토스트로 명확히 표시합니다.

## H. 사업화 & 확장

### H1. "어떤 시장에 적용 가능합니까?"

> 세 가지 시나리오를 생각했습니다.
> 첫째, 대학·학원의 LMS 보조 도구 — 강의자료 업로드 시 자동으로 학습 자료가 생성됩니다.
> 둘째, 기업 사내 교육 — 보안상 외부 LLM 못 쓰는 환경에 온프레미스 배포 가능합니다.
> 셋째, 개인 학습 도구 SaaS — 인강 영상 자동 생성이 차별화 요소입니다.

### H2. "운영 비용은 어느 정도입니까?"

> 클라우드 LLM API 를 안 쓰기 때문에 사용자당 변동비가 거의 없습니다.
> 고정비는 — Ollama 서버 GPU 1대 (RTX 4090 정도) 면 동시 사용자 수십명까지 커버 가능합니다.
> 스토리지는 학생당 인덱스 수 MB + 영상 수십 MB 정도라 미미합니다.

### H3. "기존 GPTs 같은 서비스와 어떻게 차별화됩니까?"

> 세 가지입니다.
> 온프레미스 — 강의자료가 외부로 나가지 않습니다.
> 학생 수준 자동 분류 — 같은 질문에도 학생에 맞춰 답변이 달라집니다.
> 인강 영상 자동 생성 — GPTs 에는 없는 기능입니다.

### H4. "라이선스는 어떻게 되나요? 상업적 이용 가능합니까?"

> 사용한 라이브러리는 모두 Apache 2.0 또는 MIT 계열로 상업적 이용 가능합니다.
> Qwen3 도 Apache 2.0 라이선스라 자유롭게 배포·변경 가능합니다.
> BGE-m3 도 MIT 입니다.
> 외부 LLM API 의존이 없어서 — 사용자가 늘어도 라이선스 비용 증가 없습니다.

### H5. "다른 도메인 (의료, 법률 등) 으로 확장 가능합니까?"

> 가능합니다.
> RAG 기반이라 — 도메인 자료만 바꾸면 자동으로 적응합니다. 코드 변경 없습니다.
> 다만 도메인 특화 용어가 많으면 시스템 프롬프트와 모델 선택은 도메인에 맞춰 조정이 필요합니다. 의료라면 의료 코퍼스 학습된 모델, 법률이면 법률 모델 — 같은 식입니다.

## I. 한계 & 다음 단계

### I1. "현재 가장 큰 기술 부채는?"

> 두 가지입니다.
> 첫째, 테스트 코드가 0개입니다. 졸업 프로젝트 일정상 작성하지 못했습니다. 운영 단계로 가려면 pytest 기반 단위 테스트가 필수입니다.
> 둘째, 개인화 LoRA 자동 학습 트리거가 코드만 있고 실제 자동화는 안 되어 있습니다.

### I2. "다음에 무엇을 할 계획입니까?"

> 단기적으로는 운영 인증 강화와 다중 사용자 부하 테스트입니다.
> 중장기적으로는 Kubeflow KFP 파이프라인으로 영상 생성을 병렬화하고, Kserve 로 LLM 을 분리해서 오토스케일링을 적용하는 것입니다.

### I3. "성능 모니터링은 어떻게 하실 계획입니까?"

> Prometheus + Grafana 가 자연스러운 선택입니다.
> FastAPI 측 메트릭 (요청 수, 응답 시간, 에러율) 과 Ollama 측 메트릭 (모델별 토큰 처리량, GPU 사용률) 을 같이 보고 싶습니다.
> 현재는 단순 로그 파일 기준이라 — 운영 단계에서 필수 작업입니다.

### I4. "장애가 발생하면 어떻게 복구합니까?"

> 학생별 RAG 인덱스와 영상은 디스크에 영속화되어 있어서 — 서버 재시작 시 자동 복구됩니다.
> SQLite DB 도 파일 단위라 백업/복원이 단순합니다.
> 단일 인스턴스라 고가용성 (HA) 은 아직 없습니다 — 운영 단계에서 Kubernetes 다중 replica + DB 외부화가 다음 단계입니다.

# 발표 시 마음가짐

- **모르는 질문은 솔직히** "그 부분은 검증 안 했습니다, 다음 단계 과제입니다" — 추측 금지.

- **부정적 질문에도 차분히** — "왜 이걸 안 했나요" 류는 "리소스/시간 제약 + 우선순위 판단" 으로 답.
- **마무리는 다음 단계로** — 진취적으로 보임.
