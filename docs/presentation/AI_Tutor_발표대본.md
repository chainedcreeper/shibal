## Slide 6 · 어떤 모델을 썼나 (1분)

> "모델은 LLM 에서 두 가지 선택지, 그리고 임베딩과 음성 합성용 모델까지 총 네 종류입니다.
>
> **기본 LLM 은 Qwen3 8B** 입니다. Alibaba 의 다국어 LLM 으로 약 5~6GB VRAM 만 차지합니다. 빠른 응답이 가능해서 일반 강의자료에는 8B 만으로 충분합니다.
>
> **고급 탐색용으로 Qwen3 32B** 를 별도 선택지로 두었습니다. 전공 깊이가 있거나 추론이 복잡한 자료에서는 사용자가 사이드바 드롭다운으로 32B 로 전환할 수 있습니다. 동일 코드, 같은 인프라에서 모델만 갈아끼우는 구조입니다.
>
> **임베딩은 BGE-m3** 입니다. 다국어 SOTA 모델로 약 2.3GB 입니다. 강의자료 청크를 벡터로 변환해 의미 검색을 가능하게 합니다.
>
> **재순위는 BGE Reranker v2-m3** 입니다. 초기에 20개 후보를 뽑고, 이 Reranker 로 정밀 평가한 후 관련도 최상위 3개만 선택합니다.
>
> **TTS 는 Edge-TTS** 의 한국어 SunHi 보이스를 씁니다. Microsoft 무료 API 이고, 강의 스크립트를 자연스러운 한국어 음성으로 변환합니다.
>
> 이렇게 모델들이 각자의 역할로 조합되어 — **전체 시스템이 외부 LLM API 한 번도 호출하지 않고 자체 인프라에서 완결**됩니다."

# 예상 질문 & 답변

## A. 어떻게 구현했나 (Implementation)

### A1. "데이터 격리는 정확히 어떻게 구현하셨나요?"

> JWT 토큰 안에 student_id 가 박혀있고, 모든 API 엔드포인트가 `Depends(get_current_user)` 의존성을 거치면서 토큰을 검증하고 student_id 를 추출합니다.
> 그 student_id 가 모든 DB 쿼리의 WHERE 절에 자동으로 들어갑니다. 예를 들어 `DELETE FROM notes WHERE id=? AND student_id=?` 같이 — 본인 토큰의 student_id 와 일치하는 행만 영향을 받습니다.
> RAG 인덱스도 디스크 레벨에서 `rag_indexes/<student_id>/` 디렉토리로 분리되어 있고, 영상 파일도 파일명에 student_id 가 박혀있어서 다른 ID 로는 path 자체가 매칭되지 않습니다.

### A2. "RAG 파이프라인은 어떻게 구성하셨나요?"

> 4단계입니다.
> 강의자료가 업로드되면 Parent 1000자 / Child 300자로 청킹합니다. 큰 단위는 컨텍스트 보존용, 작은 단위는 정밀 검색용입니다.
> Child 청크들을 BGE-m3 로 임베딩해서 FAISS IndexFlatIP 에 저장합니다. 정규화된 벡터의 내적이라 코사인 유사도와 동등합니다.
> 질의가 들어오면 먼저 FAISS 로 k=20 후보를 뽑고, BGE Reranker v2-m3 로 정밀 재순위해서 상위 3개를 선택합니다.
> 마지막으로 그 3개에 대응되는 Parent 청크들을 컨텍스트로 LLM 에 전달합니다.

### A3. "인강 영상은 어떻게 자동으로 생성하시나요?"

> 5단계 파이프라인입니다.
> 강의자료 텍스트가 RAG 컨텍스트로 들어가면, qwen3:32b 가 슬라이드별로 JSON 배열을 생성합니다. 각 슬라이드는 제목, 헤드라인, 불릿 포인트, 나레이션 4개 필드를 가집니다.
> 그 다음 Edge-TTS 의 한국어 SunHi 보이스로 슬라이드별 음성을 만듭니다. 단어 단위 타임스탬프도 함께 받아서 자막 burn-in 에 씁니다.
> PIL 로 슬라이드 이미지를 렌더링하고, ffmpeg concat demuxer 로 오디오와 비디오를 합칩니다. mov_text 자막을 입혀서 최종 mp4 가 완성됩니다.

### A4. "학생 수준 분류는 어떻게 동작합니까?"

> 두 단계입니다.
> 학생이 채팅을 보낼 때마다 LLM 이 그 발화를 5점 척도로 점수화합니다. "전문 용어를 정확히 사용했는가, 깊이 있는 질문인가" 같은 기준입니다.
> 그 점수가 누적되면 RandomForest 분류기가 입문, 중급, 심화 세 단계 중 하나로 분류합니다.
> 분류 결과에 따라 시스템 프롬프트가 달라집니다. 입문이면 "비유와 쉬운 예시로 설명하라", 심화면 "원리와 메커니즘까지 다루라" 같이 — 같은 질문에도 답변 스타일이 학생에 맞춰집니다.

### A5. "시험 인터랙티브 카드 UI 는 어떻게 만드셨나요?"

> 프론트엔드 측에서 LLM 응답을 파싱해서 카드로 렌더링합니다.
> 응답이 `[객관식 5문제]`, `[주관식 5문제]` 라벨로 구분되어 있고, 각 문제는 `1. 문제 내용 ① 보기1 ② 보기2 ...` 형식입니다.
> 정규식으로 라벨을 찾고 각 문제 블록을 추출한 다음, 객관식이면 ①②③④⑤ 보기를 라디오 버튼처럼 렌더링합니다.
> LLM 응답 형식이 변동 가능하기 때문에 — 마크다운 강조 `**[객관식]**`, `1)` 형식 보기, 라벨이 없는 응답까지 — 여러 fallback 을 두어 강건하게 파싱합니다.

### A6. "영상 자동 생성은 사용자가 따로 트리거 안 해도 되나요?"

> 네, 자료 업로드 직후 자동으로 시작됩니다.
> `/upload` 가 성공하면 프론트엔드가 `startVideoBackground()` 를 호출해서 `/video/generate` 를 POST 합니다. 영상 탭으로 전환하지는 않고 — 사용자가 분석 결과를 보는 동안 백그라운드로 진행됩니다.
> 2초마다 `/video/status` 를 폴링해서 진행 상황을 받고, 완료되면 토스트 알림으로 "X분 X초 소요" 라고 알려줍니다.

### A7. "환경별로 다른 모델을 쓸 수 있다고 하셨는데 어떻게 구현하셨나요?"

> 두 가지 방법으로 가능합니다.
> 첫째, 환경변수 `OLLAMA_MODEL` 로 기본 모델을 지정합니다. 서버 시작 시 그 모델로 동작합니다.
> 둘째, 런타임에 사이드바 드롭다운으로 전환 가능합니다. `POST /model` 엔드포인트가 Ollama 에 설치된 모델 목록 중에서만 선택을 허용하고, 모듈 레벨 변수를 동적으로 변경합니다. 그 다음 LLM 호출부터 새 모델이 적용됩니다.
> 코드 한 줄도 안 바뀝니다.

## B. 어떤 어려움이 있었나 (Challenges)

### B1. "개발 중 가장 까다로웠던 문제는 무엇이었습니까?"

> 새 자료를 업로드해도 화면에 옛 영상이 계속 표시되는 문제였습니다. 강력 새로고침과 캐시 삭제로도 해결되지 않았습니다.
> 원인이 세 군데였습니다. 백엔드 path 매칭, 캐시 헤더, 그리고 프론트엔드 blob 메모리 잔재.
> 그래서 3중 방어로 막았습니다 — 영상 파일명에 학생 ID 와 강의자료명을 박아서 path 자체를 다르게 만들고, 응답 URL 에 mtime 기반 cache-buster 를 추가하고, video 태그에 blob 대신 URL 을 직접 박았습니다.
> 결과적으로 옛 영상은 디스크에 보관되지만 화면에는 새 영상만 표시되고, 같은 자료를 재업로드하면 옛 영상이 자동으로 재활용되는 깔끔한 동작이 됐습니다.

### B2. "KD 학습을 폐기하셨다고 하셨는데, 그 결정은 어떻게 내리셨나요?"

> 데이터를 990개 생성한 시점에서 평가 결과를 봤습니다.
> 베이스 모델 평균 7.76점 대비 파인튜닝 후 7.97점 — 0.21점 상승했습니다.
> 그런데 새로 망가진 케이스가 13개 있었습니다. 답변이 잘리거나, 무한 반복되거나, 질문을 echo 하는 패턴이었습니다.
> 분석해보니 KorQuAD 단답형 데이터에 과적합되면서 long-form 응답 능력을 잃은 것이었습니다. catastrophic forgetting.
> DPO 같은 후처리로 고칠 수 있는 문제가 아니라 — 학습 방식 자체를 폐기하고, 같은 시간을 RAG 와 시스템 프롬프트 강화에 투자했습니다. ROI 측면에서 압도적으로 더 효과적이었습니다.

### B3. "vLLM 호환성 문제는 어떻게 해결하셨나요?"

> Kubeflow 컨테이너에 sudo 권한이 없어서 gcc 를 설치할 수 없었습니다. vLLM 의 V1 엔진이 Triton JIT 컴파일을 시도하면서 gcc 가 필요하다고 실패했습니다.
> 우회책으로 `VLLM_USE_V1=0` 환경변수로 V0 엔진을 강제하고, `enforce_eager=True` 옵션으로 컴파일 자체를 우회했습니다.
> 결과적으로 데이터 생성 파이프라인이 정상 동작했습니다.

### B4. "라이브러리 충돌은 어떻게 처리하셨나요?"

> 가장 큰 건 bcrypt 5.x 와 passlib 1.7.x 호환성 문제였습니다. 회원가입 시 500 에러가 났습니다.
> 원인은 bcrypt 5.x 에서 72바이트 stub 테스트가 truncate 안 하고 ValueError 를 던지는 변경이었습니다. passlib 가 그걸 처리하지 못했습니다.
> `requirements.txt` 에 `bcrypt<4.1` 로 버전을 고정해서 해결했습니다.
> rhwp-python 의 FreeType 충돌도 있었는데 — 컨테이너의 libfreetype 이 너무 옛 버전이어서 — 순수 파이썬 라이브러리 (python-hwpx, pyhwp) 로 폴백해서 해결했습니다.

### B5. "Python 버전 호환성 이슈도 있었다고 하셨는데?"

> `str | None` 같은 PEP 604 타입 힌트를 썼는데 Python 3.10 부터 지원되는 문법입니다.
> Conda base 환경이 3.9 였기 때문에 SyntaxError 가 나면서 서버가 부팅 자체를 못 했습니다.
> 타입 힌트를 제거하는 hotfix 로 호환성을 회복했습니다. 런타임 동작은 동일합니다.
> 이후로는 Python 3.9+ 모두에서 동작하는 코드만 쓰도록 주의하고 있습니다.

### B6. "큰 LLM 모델이 영어 reasoning 을 노출하는 문제는 어떻게 처리하셨나요?"

> qwen3:30b-a3b 같은 MoE 모델에서 `think: false` 옵션을 무시하고 영어 사고 과정을 응답 앞에 그대로 노출하는 문제가 있었습니다.
> 3중 방어로 처리했습니다.
> 첫째, 시스템 프롬프트에 "한국어로만, 사고 과정 노출 금지" 를 명시했습니다.
> 둘째, user 메시지 끝에 `/no_think` — qwen3 시리즈가 인식하는 매직 워드를 추가했습니다.
> 셋째, 클라이언트에서 stream 받을 때 영어 reasoning prefix 를 자동 필터링하는 함수를 추가했습니다. 첫 한국어 글자나 형식 마커를 찾기 전까지 버퍼링합니다.

### B7. "영상 생성에서 moviepy 가 죽는 문제는 어떻게 처리하셨나요?"

> 빈 narration 슬라이드에서 Edge-TTS 가 빈 오디오를 반환했고, moviepy 가 그걸 합치다가 zero-size array 에러로 죽었습니다.
> 세 가지 가드를 박았습니다 — 빈 narration 은 슬라이드 title 로 폴백, ffprobe 로 실제 audio duration 을 측정, 그래도 0초면 1초로 패딩.
> 또한 moviepy 자체가 너무 느려서 (14분 소요) 결국 ffmpeg concat demuxer 로 전환했습니다. 같은 작업이 2분 16초로 단축됐습니다.

## C. 기술 선택

### C1. "왜 기본 모델로 8B 를 선택하셨나요? 32B 와의 사용 기준은?"

> 운영 효율과 응답 속도 때문입니다.
> 일반 강의자료에서는 RAG 가 컨텍스트를 정확히 잡아주기 때문에, 8B 만으로도 응답 품질이 충분합니다. 토큰 생성 속도도 8B 가 32B 보다 약 4배 빠르고 — 사용자 체감이 훨씬 좋습니다.
> 32B 는 어려운 자료 — 전공 깊이가 있거나 복잡한 추론이 필요한 경우 — 를 위한 **고급 탐색 모드**로 별도 선택지로 두었습니다. 사이드바 드롭다운에서 사용자가 직접 전환할 수 있습니다.
> 즉 8B 가 **속도와 효율의 기본값**, 32B 가 **품질 우선의 고급 옵션**입니다. 코드는 동일하고 `OLLAMA_MODEL` 환경변수와 드롭다운으로 즉시 전환됩니다.

### C2. "왜 RAG 인가요? 파인튜닝과 비교하면?"

> ROI 차이 때문입니다.
> 파인튜닝(KD)을 시도했는데 990개 데이터로 평균 점수 0.21점 상승, 새로 망가진 케이스 13개였습니다.
> 같은 시간을 RAG 와 시스템 프롬프트 분기에 투자하면 학습 0시간으로 동등하거나 더 좋은 결과가 나왔습니다. 도메인 지식은 RAG 에 맡기고, 모델은 스타일과 톤만 담당하는 게 효과적이었습니다.

### C3. "왜 Ollama 인가요? vLLM 이나 TGI 와 비교하면?"

> 배포 단순성 때문입니다.
> vLLM 은 Triton JIT 컴파일러와 gcc 가 필요해서 — sudo 권한 없는 환경에서는 띄울 수 없습니다.
> Ollama 는 user-space 바이너리 하나로 끝납니다. 어떤 환경에서든 동일한 명령으로 실행됩니다.
> 성능은 vLLM 이 조금 더 빠르지만, 운영 단순성이 그 차이를 상쇄합니다.

## D. 보안 / 운영

### D1. "JWT 토큰이 탈취되면 어떻게 됩니까?"

> 토큰 만료가 24시간으로 짧게 잡혀있습니다. 탈취된 토큰의 영향 범위가 시간적으로 제한됩니다.
> 또한 SECRET_KEY 가 변경되면 모든 기존 토큰이 무효화되기 때문에 — 긴급 시 운영자가 즉시 모든 세션을 끊을 수 있습니다.

### D2. "다중 사용자 환경에서 성능은 어떻습니까?"

> 단일 Ollama 인스턴스가 LLM 요청을 직렬화하기 때문에, 동시 5명 이상이면 대기 시간이 늘어납니다.
> 영상 생성은 백그라운드 잡이라 사용자 체감은 적습니다.
> 운영 규모를 키우려면 Kserve InferenceService 로 LLM 을 분리하고 오토스케일링을 적용하는 게 다음 단계입니다.

### D3. "운영 비용은 어떻게 되나요?"

> 외부 LLM API 를 호출하지 않기 때문에 사용자당 변동비가 거의 없습니다.
> 고정비는 GPU 서버 1대 정도 — RTX 4090 급이면 동시 사용자 수십명까지 커버합니다.
> 스토리지는 학생당 수십 MB 수준이라 미미합니다.

## E. 사업화 / 확장

### E1. "어떤 시장에 적용 가능합니까?"

> 세 가지 시나리오를 생각했습니다.
> 첫째, 대학·학원의 LMS 보조 도구 — 강의자료 업로드 시 자동으로 학습 자료가 생성됩니다.
> 둘째, 기업 사내 교육 — 보안상 외부 LLM 못 쓰는 환경에 온프레미스 배포 가능합니다.
> 셋째, 개인 학습 도구 SaaS — 인강 영상 자동 생성이 차별화 요소입니다.

### E2. "기존 GPTs 같은 서비스와 어떻게 차별화됩니까?"

> 세 가지입니다.
> 온프레미스 — 강의자료가 외부로 나가지 않습니다.
> 학생 수준 자동 분류 — 같은 질문에도 학생에 맞춰 답변이 달라집니다.
> 인강 영상 자동 생성 — GPTs 에는 없는 기능입니다.
