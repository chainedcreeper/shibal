## [슬라이드 2] 머신러닝과 LLM

먼저 기본 개념부터 짚겠습니다.

머신러닝은 데이터로부터 규칙을 스스로 학습하는 기술입니다.
크게 세 가지로 나뉘는데요.
정답 레이블로 학습하는 지도학습,
패턴을 스스로 찾는 비지도학습,
보상 신호로 행동을 최적화하는 강화학습이 있습니다.
딥러닝은 그 중 신경망 기반 방식으로 이미지, 음성, 텍스트 처리에 특히 강합니다.

LLM은 이 딥러닝을 텍스트에 극한까지 적용한 결과물입니다.
수천억 개의 텍스트를 학습해서 언어 자체를 이해하고 생성하는 모델입니다.
GPT-4, Claude, Gemini 같은 클로즈드 모델과
LLaMA, Qwen, Mistral 같은 오픈소스 모델로 나뉩니다.
핵심 능력은 자연어 이해와 생성이고, 다양한 태스크를 범용으로 처리할 수 있습니다.

### 예상 질문

**Q. 머신러닝이랑 AI가 같은 건가요?**
AI가 더 큰 개념입니다. AI ⊃ 머신러닝 ⊃ 딥러닝 ⊃ LLM 순서로 포함 관계가 됩니다.
AI는 "지능적으로 행동하는 시스템" 전반을 말하고, 머신러닝은 그 중 데이터로 학습하는 방식입니다.

**Q. 딥러닝이랑 머신러닝은 뭐가 다른가요?**
머신러닝은 데이터로 학습하는 방법들의 총칭이고, 딥러닝은 그 중 인공신경망을 깊게 쌓은 방식입니다.
딥러닝 이전에도 결정 트리, SVM, 랜덤 포레스트 같은 머신러닝 방법이 있었습니다.

**Q. GPT랑 LLM이 같은 말인가요?**
GPT는 LLM의 한 종류입니다. GPT는 OpenAI가 만든 특정 모델 시리즈 이름이고,
LLM은 이런 초거대 언어 모델 전체를 부르는 일반 명칭입니다.

**Q. 오픈소스 모델과 클로즈드 모델 차이가 뭔가요?**
클로즈드 모델(GPT-4, Claude 등)은 API로만 사용 가능하고 내부 구조가 공개되지 않습니다.
오픈소스 모델(LLaMA, Qwen 등)은 가중치와 구조가 공개되어 직접 다운받아 실행하거나 수정할 수 있습니다.
오픈소스는 비용이 없는 대신 직접 서버를 운용해야 합니다.

**Q. 파라미터가 많으면 무조건 좋은 건가요?**
꼭 그렇지는 않습니다. 파라미터가 많을수록 표현력이 높아지지만 추론 속도가 느리고 메모리를 많이 씁니다.
최근에는 학습 데이터와 방법론을 개선해서 작은 모델로도 큰 성능을 내는 방향으로 발전하고 있습니다.
예를 들어 Qwen3-8B가 예전 70B 모델 수준의 성능을 냅니다.

## [슬라이드 3] LLM 동작 구조

LLM이 어떻게 동작하는지 구조를 간략히 설명드리겠습니다.

LLM의 핵심 구조는 Transformer입니다.
2017년 Google이 발표한 구조로, Self-Attention이라는 메커니즘으로 입력 토큰들 간의 관계를 학습합니다.
기존 RNN 방식은 문장을 순서대로 처리해서 앞의 내용을 잊어버리는 문제가 있었는데,
Transformer는 모든 토큰을 동시에 보면서 어느 토큰이 어느 토큰과 관련 있는지 계산합니다.
레이어를 쌓을수록 더 추상적인 개념을 표현할 수 있고, 7B 모델 기준으로 약 32개 레이어가 있습니다.

학습은 사전학습 단계에서 다음 토큰 예측을 수조 번 반복하면서 언어 자체를 습득합니다.
여기에 수백만 달러, 수개월이 들어가고 GPT나 LLaMA 같은 베이스 모델이 만들어집니다.

추론 시에는 텍스트를 토큰이라는 숫자 단위로 변환하고, 토큰을 하나씩 순서대로 생성합니다.
temperature 값으로 얼마나 창의적으로 답변할지 조절할 수 있습니다.

### 예상 질문

**Q. Self-Attention이 정확히 뭔가요?**
각 토큰이 다른 모든 토큰에 대해 "얼마나 관련이 있는가"를 점수로 계산하는 메커니즘입니다.
예를 들어 "그 고양이는 배가 고팠다"에서 "그"가 "고양이"와 높은 점수를 가지면, "그"를 처리할 때 "고양이" 정보를 많이 참고합니다.
이걸 통해 멀리 떨어진 단어도 연결할 수 있습니다.

**Q. 토큰이 뭔가요?**
LLM이 텍스트를 처리하는 최소 단위입니다.
영어는 보통 단어나 단어 조각으로 나뉘고, 한국어는 글자나 음절 단위로 나뉩니다.
"안녕하세요"는 약 5~8개 토큰으로 변환될 수 있습니다.
한국어가 영어보다 토큰 수가 많이 나와서 같은 내용도 비용이 더 드는 경향이 있습니다.

**Q. temperature가 뭔가요?**
생성할 때 얼마나 무작위성을 넣을지 결정하는 값입니다.
0에 가까울수록 항상 가장 높은 확률의 토큰을 선택(결정적), 높을수록 낮은 확률의 토큰도 선택(창의적).
사실 확인이 중요한 경우는 낮게, 창의적인 글쓰기는 높게 설정합니다.

**Q. 7B, 14B 같은 숫자가 뭔가요?**
파라미터(가중치) 수입니다. B는 Billion(10억)의 약자.
7B = 70억 개 파라미터. 숫자가 클수록 표현력이 높고 VRAM을 많이 씁니다.
7B는 약 16GB, 14B는 약 32GB VRAM이 필요합니다(fp16 기준).

**Q. 컨텍스트 윈도우가 뭔가요?**
모델이 한 번에 처리할 수 있는 최대 토큰 수입니다.
컨텍스트 윈도우를 넘어서면 앞의 내용을 잊어버립니다.
최근 모델들은 128K, 심지어 1M 토큰까지 지원하기도 합니다.

## [슬라이드 4] LLM 활용 & 성능 향상 방법

LLM 성능을 높이거나 활용하는 방법이 크게 다섯 가지 있습니다.

첫 번째는 프롬프트 엔지니어링입니다.
시스템 프롬프트, Few-shot 예시, Chain-of-Thought 유도 같은 방식으로
비용 없이 성능을 끌어올릴 수 있습니다.
다만 모델 능력 자체가 바뀌는 건 아니라서 한계가 있습니다.

두 번째는 RAG입니다.
질문이 들어오면 관련 문서를 검색해서 컨텍스트로 넘겨주는 방식입니다.
최신 정보 반영과 할루시네이션 감소에 효과적이지만 검색 실패 시 취약합니다.

세 번째는 파인튜닝입니다.
도메인 데이터로 모델 자체를 추가 학습하는 방식으로,
LoRA나 QLoRA를 쓰면 개인 GPU에서도 가능합니다. 데이터 품질이 전부입니다.

네 번째는 에이전트입니다.
LLM이 웹 검색, 코드 실행, DB 조회 같은 툴을 직접 사용하게 하는 방식으로
LangChain, AutoGen 같은 프레임워크가 있습니다.

다섯 번째는 지식 증류입니다.
큰 모델의 추론 능력을 작은 모델에 이식하는 방식으로
DeepSeek-R1, Qwen3 distill이 이 방식으로 만들어졌습니다.

### 예상 질문

**Q. 할루시네이션이 뭔가요?**
LLM이 사실이 아닌 내용을 그럴듯하게 지어내는 현상입니다.
모델이 학습 데이터에서 패턴을 익히다 보니, 실제로 존재하지 않는 논문 제목이나 사람을 만들어내기도 합니다.
RAG나 Faithfulness 학습으로 줄일 수 있습니다.

**Q. RAG랑 파인튜닝 중에 뭐가 더 좋나요?**
목적이 다릅니다. 둘은 경쟁 관계가 아니라 보완 관계입니다.
RAG는 "최신 정보나 특정 문서 기반 답변"에 적합하고,
파인튜닝은 "모델의 말투, 형식, 특정 도메인 능력"을 바꾸는 데 적합합니다.
할루시네이션 문제는 둘 다 써야 효과적입니다.

**Q. Chain-of-Thought가 뭔가요?**
"단계적으로 생각해봐" 같은 프롬프트로 모델이 추론 과정을 거치게 하는 방식입니다.
"1+1+1+1+1이 뭐야?"보다 "하나씩 더해봐. 1+1=2, 2+1=3, 3+1=4, 4+1=?" 처럼 유도하면 더 정확해집니다.

**Q. 에이전트랑 챗봇은 다른 건가요?**
챗봇은 질문에 답변하는 것에 집중하고, 에이전트는 목표를 달성하기 위해 여러 단계를 스스로 계획하고 실행합니다.
예를 들어 "다음 주 날씨 보고 여행 일정 짜줘"라고 하면 에이전트는 날씨 API 호출 → 일정 생성 → 숙소 검색까지 스스로 처리합니다.

**Q. LoRA가 뭔가요?**
Low-Rank Adaptation. 모델 전체를 학습하는 대신 작은 행렬 두 개만 추가해서 학습하는 방식입니다.
전체 파라미터의 약 0.3%만 학습해서 VRAM을 대폭 절약합니다.
7B 모델 기준 Full Fine-tuning은 약 112GB가 필요하지만, LoRA는 20~30GB면 됩니다.

**Q. QLoRA는 LoRA랑 뭐가 다른가요?**
LoRA에 4비트 양자화를 더한 방식입니다.
베이스 모델 자체를 4비트로 압축해서 메모리를 절반으로 줄이고, LoRA 어댑터만 bf16으로 학습합니다.
7B 모델 기준 약 10GB VRAM으로 학습이 가능해서 RTX 3090 한 장으로도 됩니다.

**Q. 지식 증류에서 Teacher, Student가 뭔가요?**
Teacher는 크고 성능 좋은 모델, Student는 학습시킬 작은 모델입니다.
Teacher가 먼저 데이터에 대한 답변(추론 과정 포함)을 생성하고,
Student가 그 결과를 보고 따라 학습합니다.
결과적으로 Student가 Teacher에 가까운 능력을 갖추게 됩니다.

## [슬라이드 5] 라이브러리 & 도구 생태계

실제로 쓰는 도구들을 간략히 소개드리겠습니다.

학습 쪽에서는 HuggingFace의 PEFT가 LoRA 구현의 표준이고,
TRL이 SFT, DPO 같은 다양한 학습 방식을 통합 지원합니다.
Unsloth는 LoRA 학습을 2~5배 빠르게 해주고,
Axolotl은 YAML 설정 파일 하나로 파인튜닝을 할 수 있게 해줍니다.

서빙 쪽에서는 llama.cpp가 모델을 GGUF 포맷으로 양자화해서 경량 실행을 가능하게 하고,
Ollama가 이를 활용해서 로컬 LLM 서버를 1분 만에 띄울 수 있게 해줍니다.
고성능이 필요하면 vLLM, GUI로 편하게 테스트하고 싶으면 LM Studio를 씁니다.

에이전트 프레임워크는 LangChain이 가장 널리 쓰이고,
문서 기반 RAG는 LlamaIndex, 멀티 에이전트는 AutoGen이 대표적입니다.

### 예상 질문

**Q. HuggingFace가 뭔가요?**
AI 모델과 데이터셋을 공유하는 플랫폼이자 관련 라이브러리를 만드는 회사입니다.
GitHub처럼 누구나 모델을 올리고 내려받을 수 있어서 오픈소스 AI 생태계의 중심입니다.
transformers, PEFT, TRL, datasets 등 핵심 라이브러리를 만듭니다.

**Q. GGUF가 뭔가요?**
GPU-General Unified Format. 모델 가중치를 양자화해서 저장하는 포맷입니다.
q4_k_m은 4비트 양자화 중 균형이 좋은 방식으로 가장 많이 씁니다.
용량이 절반 이하로 줄고 CPU에서도 실행 가능합니다.

**Q. Ollama랑 vLLM 차이가 뭔가요?**
Ollama는 설치 1분, 사용 단순, 개발/개인용에 적합합니다.
vLLM은 PagedAttention으로 처리량이 훨씬 빠르고 서버 배포에 적합합니다.
둘 다 OpenAI API 호환이라 코드 한 줄만 바꾸면 전환됩니다.

**Q. LangChain이 뭔가요?**
LLM 기반 애플리케이션을 만들기 위한 프레임워크입니다.
프롬프트 관리, RAG 구성, 에이전트 툴 연결, 메모리 관리 같은 기능을 쉽게 조합할 수 있습니다.
"LLM을 연결해서 뭔가 만들고 싶다"면 대부분 LangChain이나 LlamaIndex로 시작합니다.

**Q. GPU 없으면 아무것도 못 하나요?**
학습은 GPU가 사실상 필수입니다.
하지만 추론(서빙)은 llama.cpp + GGUF로 CPU에서도 가능합니다(느리지만).
API를 쓰면 로컬 GPU 없이 GPT-4, Claude 수준의 모델을 쓸 수 있습니다.
학습이 필요하면 클라우드 GPU(RunPod, Lambda Labs)를 시간당 3~5달러에 빌릴 수 있습니다.

**Q. SFT가 뭔가요?**
Supervised Fine-Tuning. 정답 입출력 쌍으로 지도학습하는 가장 기본적인 파인튜닝 방식입니다.
"이런 입력이 들어오면 이런 출력을 해라"를 직접 학습시킵니다.
파인튜닝의 첫 단계로 보통 SFT → DPO 또는 SFT → RLHF 순서로 진행합니다.

**Q. DPO가 뭔가요?**
Direct Preference Optimization. 좋은 답변과 나쁜 답변 쌍을 줘서 선호를 직접 학습합니다.
기존 RLHF는 보상 모델을 따로 학습하고 강화학습을 돌려야 해서 복잡했는데,
DPO는 그 과정을 단순화해서 SFT처럼 바로 학습이 됩니다. 최근 가장 많이 씁니다.

## [슬라이드 6] AI Tutor

이제 제가 실제로 만들고 있는 것을 간단히 소개드리겠습니다.

대학 강의자료 AI Tutor입니다.
FastAPI 백엔드에 FAISS 벡터 DB와 BGE Reranker로 RAG를 구성하고,
Qwen3-8B를 Ollama로 로컬에서 서빙하는 구조입니다.

기본 구현은 돼 있었는데 두 가지 문제가 있었습니다.
지문에 없는 내용을 지어내는 할루시네이션이랑
단계적 설명 없이 단답만 하는 문제였습니다.
그래서 Faithfulness를 높이고 CoT 설명이 가능하게 만드는 게 목표입니다.

### 예상 질문

**Q. RAG는 어떻게 동작하나요?**
크게 세 단계입니다.
① PDF 등 문서를 청크로 나눠서 임베딩(벡터)으로 변환해 DB에 저장
② 질문이 들어오면 질문도 임베딩으로 변환해서 유사한 청크를 검색
③ 검색된 청크를 컨텍스트로 LLM에 전달해서 답변 생성
FAISS는 이 중 ②③ 사이의 유사도 검색 엔진입니다.

**Q. BGE Reranker가 뭔가요?**
검색된 결과를 재순위하는 모델입니다.
벡터 검색으로 후보 문서 20개를 뽑고, Reranker가 질문과의 관련성을 다시 정밀하게 평가해서 상위 3~5개만 LLM에 넘깁니다.
검색 품질이 올라가서 할루시네이션을 줄이는 데 도움이 됩니다.

**Q. FAISS가 뭔가요?**
Facebook AI Research에서 만든 벡터 유사도 검색 라이브러리입니다.
수백만 개의 벡터 중에서 가장 비슷한 벡터를 빠르게 찾아줍니다.
RAG에서 문서 검색 엔진 역할을 합니다.

**Q. Faithfulness가 뭔가요?**
모델이 지문에 있는 내용만 답변하는지 측정하는 지표입니다.
생성한 답변의 단어 중 지문에 있는 단어 비율로 계산합니다.
이 값이 낮으면 지문을 무시하고 내용을 지어냈다는 뜻입니다.

**Q. CoT가 뭔가요?**
Chain-of-Thought. 단계적으로 추론하는 과정을 생성하게 만드는 방식입니다.
"답변은 A입니다"가 아니라 "왜냐하면 ~이고, 따라서 ~이므로, 결론적으로 A입니다"처럼
추론 과정이 포함된 답변을 생성하게 합니다.

## [슬라이드 7] Reasoning Distillation

접근 방식은 Reasoning Distillation입니다.

Wikipedia 한국어 지문을 Qwen3-14B에게 주고 CoT QA를 생성시킨 다음,
KLUE/MRC와 OpenThoughts를 병합해서 약 50,000개 데이터셋을 구성했습니다.
그걸로 Qwen3-8B를 LoRA r=32로 학습시키는 구조입니다.

학습 도구는 Unsloth와 TRL 조합을 썼고,
학교 Kubeflow GPU 서버에서 돌리고 있습니다.
평가는 ROUGE-L, BERTScore, Faithfulness, Hallucination Rate로 합니다.

### 예상 질문

**Q. 왜 Qwen3를 선택했나요?**
세 가지 이유입니다.
한국어 성능이 오픈소스 중 상위권이고,
thinking 모드(<think> 태그)를 공식 지원해서 CoT 데이터 생성과 학습에 바로 연결되고,
Apache 2.0 라이선스라 제약이 없습니다.

**Q. Teacher 모델로 GPT-4나 Claude를 쓰면 더 좋지 않나요?**
맞습니다. Teacher가 강할수록 데이터 품질이 올라갑니다.
다만 OpenAI ToS는 API 출력으로 모델 학습 데이터를 만드는 걸 제한합니다.
그래서 라이선스 문제 없고 비용도 들지 않는 Qwen3-14B를 로컬에서 돌렸습니다.

**Q. 합성 데이터로 학습하는 게 효과가 있나요?**
있습니다. DeepSeek-R1, Qwen3 distill 모두 합성 데이터로 만들어진 모델입니다.
완벽하진 않지만 비용 없이 대량 데이터를 확보할 수 있다는 장점이 큽니다.
품질 필터링을 잘 하는 게 핵심입니다.

**Q. ROUGE-L이 뭔가요?**
Recall-Oriented Understudy for Gisting Evaluation. 텍스트 생성 품질 지표입니다.
생성 텍스트와 정답 텍스트 사이의 최장 공통 부분 수열을 기반으로 F1 점수를 계산합니다.
빠르고 계산이 쉽지만 의미보다 단어 겹침만 보는 한계가 있습니다.

**Q. BERTScore가 뭔가요?**
BERT 임베딩 기반 의미 유사도 지표입니다.
단순 단어 겹침이 아니라 문맥적 의미를 비교해서 동의어나 패러프레이즈도 인식합니다.
ROUGE보다 정확하지만 계산 비용이 높습니다.

**Q. rank r=32로 정한 이유가 뭔가요?**
태스크 복잡도에 따라 정합니다.
단순 분류나 요약은 r=8~16으로 충분하고,
CoT 추론처럼 복잡한 패턴을 학습해야 하면 r=32~64를 씁니다.
r이 작으면 underfitting, 너무 크면 VRAM을 더 쓰기 때문에 균형을 잡아야 합니다.

**Q. Kubeflow가 뭔가요?**
Kubernetes 기반 ML 워크플로우 오케스트레이션 플랫폼입니다.
GPU 자원 스케줄링, 다중 실험 관리, 파이프라인 자동화를 지원합니다.
학교 GPU 서버에 설치된 환경을 그대로 활용했습니다.

## [슬라이드 8] 현재 & Q&A

파이프라인과 데이터셋 준비는 완료됐고, 지금 학습이 돌아가고 있습니다.
결과 나오면 서빙에 연결하고, 좋으면 DPO 정렬까지 이어갈 계획입니다.

이상입니다. 질문 받겠습니다.

### 예상 질문

**Q. 결과가 없는데 왜 발표를 했나요?**
3주차 중간 공유 자리라 결과보다 접근 방식과 생태계 소개가 목적이었습니다.
결과는 다음 주에 공유하겠습니다.

**Q. 결과가 안 좋으면 어떻게 할 건가요?**
원인을 두 가지로 나눠서 볼 계획입니다.

학습 설정 문제라면 rank, 에폭, 학습률 조정입니다.

**Q. 4주 안에 끝낼 수 있나요?**
파인튜닝 학습과 서빙 연결은 4주 안에 가능합니다.
DPO 정렬은 결과에 따라 추가 진행 여부를 결정합니다.
