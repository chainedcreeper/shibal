## [슬라이드 2] 파인튜닝이란?

저희 AI Tutor 프로젝트에서 LLM을 쓰다 보니 몇 가지 문제가 있었습니다.
큰 모델은 느리고 서버 비용 부담이 있었고, 작은 모델은 지문에 없는 내용을 지어내는 할루시네이션 문제가 있었습니다.

이걸 해결하는 방법이 파인튜닝입니다.
파인튜닝은 이미 언어를 아는 사전학습된 LLM을 우리 목적에 맞게 추가 학습시키는 겁니다.

방법은 두 가지를 조합했습니다.
첫 번째는 Knowledge Distillation, 지식 증류입니다.
큰 Teacher 모델의 추론 과정을 작은 Student 모델에 이식하는 방식입니다.
두 번째는 LoRA입니다.
모델 전체를 학습하는 게 아니라 내부에 작은 보정 행렬 두 개만 추가해서 학습하는 방식으로, 48GB VRAM으로도 충분히 할 수 있습니다.

## [슬라이드 3] KD + LoRA 상세

Knowledge Distillation을 좀 더 설명드리겠습니다.

저희가 쓰는 방식은 Reasoning Distillation입니다.
Teacher인 Qwen3-14B가 지문을 보고 추론 과정이 담긴 QA를 생성하면,
그걸 Student인 Qwen3-8B가 학습합니다.
단순히 정답만 주는 게 아니라 think 태그 안의 추론 과정까지 학습시키는 겁니다.
(DeepSeek-R1이나 Qwen3 distill 같은 모델들도 이 방식으로 만들어졌습니다.)

LoRA는 쉽게 말하면 이렇습니다.
원본 악보는 그대로 두고, 포스트잇에 "이 부분은 이렇게 연주해라"고 메모만 붙이는 것과 같습니다.
연주할 때는 원본 악보와 포스트잇을 같이 보는 거고, 우리는 포스트잇 부분만 학습하는 겁니다.

이걸 수식으로 표현하면 W' = W₀ + B·A입니다.
W₀가 원본 악보, B·A가 포스트잇입니다.
포스트잇을 두 장으로 쪼개서 아주 얇게 만든 게 B와 A인데, 이렇게 하면 크기가 훨씬 작아집니다.
결과적으로 전체 파라미터의 0.3%만 학습하는 셈이라 Full Fine-tuning의 112GB 대비 28GB면 충분합니다.

## [슬라이드 4] 어떻게 할 건지 + 데이터셋

전체 파이프라인은 이렇습니다.
Wikipedia 한국어 지문을 Teacher에게 주고 CoT QA를 생성하고,
거기에 추가 데이터셋들을 병합해서 약 65,000개 데이터셋을 만들고,
그걸로 Student를 LoRA 학습시킨 다음 GGUF로 변환해서 Ollama에 올리는 구조입니다.

데이터셋은 다섯 가지를 병합했습니다.

첫 번째는 Wikipedia CoT로 5,000개, Teacher가 직접 생성한 핵심 데이터입니다.
두 번째는 KorQuAD 1.0에서 10,000개, 한국어 독해 QA 벤치마크입니다.
세 번째는 KLUE/MRC에서 10,000개, 다양한 도메인의 한국어 독해 데이터입니다.
네 번째는 sae4K에서 10,000개, 네이버 뉴스 요약 데이터셋입니다.
다섯 번째는 OpenThoughts에서 30,000개, 영어 CoT 추론 트레이스 데이터입니다.
합쳐서 약 65,000개로 학습합니다.

(각 데이터셋이 왜 필요한지, 어떻게 찾았는지는 질문 오시면 설명드리겠습니다.)

학습은 Qwen3-8B에 LoRA r=32로, 에폭 2번 돌립니다.
소요 시간은 약 2~3시간으로 생각보다 짧습니다.
배포까지 하면 추가로 30분 정도면 됩니다.

## [슬라이드 5] 지금 하고 있는 것

현재 진행 상황입니다.

지금은 데이터 생성 단계로, Kubeflow에서 Teacher가 CoT QA를 만들고 있습니다.
약 13시간 걸리는 작업인데 현재 50% 정도 완료됐습니다.

(학습 환경 구성 과정에서 시행착오가 있었는데, Kubeflow 기본 환경에 이미 설치된 패키지들과 버전 충돌이 심해서 별도 가상환경을 만들어서 진행했습니다. vLLM, torch, peft, trl이 각각 요구하는 버전이 달라서 격리 환경이 필요했습니다. 거기다 vLLM 엔진 초기화 문제도 있었는데, gcc 컴파일러가 없어서 Triton JIT 컴파일이 실패하는 문제였습니다. 결국 vLLM 대신 transformers로 바꿔서 해결했습니다.)

생성이 끝나면 데이터셋 병합 30분, 학습 2~3시간, 평가 3~4시간, 배포 30분 순서로 진행합니다.
평가는 Teacher가 Student 답변을 하나하나 검증하는 방식이라 시간이 좀 걸립니다.
결과가 좋지 않으면 설정을 바꿔서 재학습도 고려하고 있습니다.

## [슬라이드 6] 평가를 어떻게 할 것인가

평가 방법은 아직 고민 중인데, 메인으로 생각하는 건 LLM-as-a-Judge 방식입니다.

Teacher인 Qwen3-14B가 채점관이 되어서 Student의 답변을 직접 평가하는 겁니다.
지문 근거를 했는지, 추론 과정이 올바른지, Teacher 답변과 얼마나 유사한지를 봅니다.
숫자 지표보다 직관적이고 도메인에 맞는 평가가 가능하다는 장점이 있습니다.

(한계도 있습니다. Teacher가 만든 데이터로 학습시켰으니 Teacher가 채점하면 편향이 있을 수 있습니다. 그래서 보조 지표도 같이 봅니다.)

보조 지표로는 텍스트 유사도인 ROUGE-L, 의미적 유사도인 BERTScore,
지문 충실도인 Faithfulness, 할루시네이션 비율도 같이 봅니다.

### 파인튜닝 관련

**Q. 파인튜닝이랑 그냥 프롬프트 잘 짜는 거랑 뭐가 달라요?**
프롬프트 엔지니어링은 모델 능력 자체는 그대로고 입력만 바꾸는 거고,
파인튜닝은 모델 자체를 변경해서 능력을 키우는 겁니다.
할루시네이션 같은 근본적인 문제는 파인튜닝으로 해결해야 합니다.

**Q. Knowledge Distillation이랑 그냥 파인튜닝이랑 뭐가 달라요?**
일반 파인튜닝은 정답 데이터로 학습하는 거고,
KD는 큰 모델의 추론 과정 자체를 가르치는 겁니다.
단순 정답보다 훨씬 풍부한 정보를 학습할 수 있습니다.

**Q. LoRA를 쓰는 이유가 뭔가요?**
GPU 메모리 제약 때문입니다. 전체 파인튜닝은 100GB 이상 필요한데
LoRA는 전체의 0.3%만 학습해서 28GB면 됩니다.
성능 차이는 거의 없으면서 훨씬 현실적입니다.

**Q. B·A 행렬 두 개로 쪼개는 게 왜 더 작아지나요?**
예를 들어 4096×4096 행렬 하나는 약 1,680만 개 파라미터입니다.
이걸 4096×32 행렬 A와 32×4096 행렬 B로 쪼개면 합쳐서 26만 개입니다.
64배 줄어드는 겁니다.
포스트잇 비유로 보면, 큰 포스트잇 하나보다 좁고 긴 포스트잇 두 장이 훨씬 작은 것과 같습니다.

**Q. 그럼 B·A를 합치면 원래 행렬 크기가 되는데, 표현력이 부족하지 않나요?**
맞습니다. 그래서 rank r을 크게 할수록 표현력이 올라가는 대신 파라미터가 늘어납니다.
저희는 r=32로 설정했는데, CoT 추론처럼 복잡한 패턴을 학습하려면 이 정도가 필요합니다.
단순한 태스크는 r=8~16으로도 충분합니다.

**Q. LoRA rank r=32는 왜 그 값으로 정했나요?**
태스크 복잡도에 따라 다른데, 단순 분류나 요약은 r=8~16으로 충분하고
CoT 추론처럼 복잡한 패턴을 학습해야 하면 r=32~64를 씁니다.
r이 작으면 표현력이 부족하고, 너무 크면 VRAM을 더 쓰기 때문에 32로 잡았습니다.

### 데이터셋 관련

**Q. Wikipedia 데이터를 왜 Teacher한테 생성시켰나요? 그냥 있는 QA 데이터 쓰면 안 되나요?**
있는 데이터도 씁니다. KorQuAD나 KLUE/MRC가 그 케이스입니다.
근데 저희 목적이 강의 AI Tutor라서 "지문 보고 단계적으로 설명하는 능력"이 중요합니다.
기존 데이터셋엔 think 태그로 추론 과정이 담긴 게 없어서 Teacher가 직접 생성했습니다.

**Q. Wikipedia에서 어떻게 데이터를 가져왔나요?**
HuggingFace에 `wikimedia/wikipedia` 데이터셋이 공개되어 있어서 한국어 버전을 스트리밍으로 불러왔습니다.
600자에서 8,000자 사이 문서만 필터링하고, 1,200자로 잘라서 지문으로 씁니다.
직접 크롤링하는 게 아니라 이미 정제된 공개 데이터셋을 그대로 쓰는 겁니다.

**Q. Wikipedia 데이터가 강의 학습에 적합한가요?**
몇 가지 이유에서 적합합니다.
첫째로 검증된 사실 기반이라 할루시네이션 줄이는 학습에 좋습니다.
둘째로 과학, 역사, 기술 등 다양한 도메인을 커버해서 강의 주제와 잘 맞습니다.
셋째로 실제로 LLaMA, Qwen 같은 대형 모델들도 Wikipedia를 주요 사전학습 데이터로 씁니다.
다만 강의 특화 데이터가 아니라는 한계는 있어서, 추후 실제 강의 자료로 보완할 계획입니다.

**Q. KorQuAD가 뭔가요?**
한국어 독해 QA 벤치마크입니다.
Wikipedia 지문과 그에 맞는 질문-답변 쌍으로 구성되어 있고 약 60,000개입니다.
KLUE/MRC와 비슷하지만 더 큰 규모입니다.

**Q. KLUE/MRC는 뭔가요?**
한국어 언어 이해 벤치마크인 KLUE의 독해 파트입니다.
뉴스, 위키백과, 법령 등 다양한 도메인의 한국어 QA 데이터셋입니다.
약 10,000개를 사용합니다.

**Q. sae4K는 왜 넣었나요?**
저희 AI Tutor의 주요 기능 중 하나가 PDF 요약입니다.
sae4K는 네이버 뉴스 문서와 요약 쌍으로 구성된 데이터라 요약 능력 학습에 적합합니다.
AwesomeKorean_Data 리포지토리에서 찾았습니다.

**Q. OpenThoughts는 영어 데이터인데 왜 넣었나요?**
추론 능력 자체는 언어에 독립적인 부분이 있습니다.
수학이나 논리 추론 방식은 한국어든 영어든 같은 패턴이라 보강에 도움이 됩니다.
실제로 Qwen3 같은 다국어 모델 학습에도 영어 추론 데이터를 많이 씁니다.

**Q. 데이터가 65,000개면 충분한가요?**
태스크에 따라 다른데 LoRA SFT 기준으로 수천~수만 개면 효과가 납니다.
중요한 건 양보다 품질입니다.
Teacher가 생성한 5,000개 CoT 데이터가 핵심이고, 나머지는 보완 역할입니다.

### 환경 관련

**Q. 왜 가상환경을 따로 만들었나요?**
Kubeflow 기본 환경에 이미 설치된 패키지들과 버전 충돌이 심했기 때문입니다.
vLLM, torch, peft, trl이 각각 요구하는 버전이 달라서 기본 환경에 설치하면 기존 패키지들이 깨집니다.
별도 가상환경으로 격리해서 충돌 없이 관리했습니다.

**Q. 가상환경이 뭔가요? 어떻게 만드나요?**
파이썬 패키지들을 시스템 전체가 아니라 특정 폴더 안에만 격리해서 설치하는 공간입니다.
프로젝트마다 독립된 패키지 창고를 만드는 것과 같습니다.

만드는 방법은 간단합니다.
Kubeflow Jupyter 터미널에서 아래 두 줄만 치면 됩니다.

  python -m venv ~/train_env       ← train_env 폴더 안에 독립된 파이썬 환경 생성
  source ~/train_env/bin/activate  ← 이 환경을 활성화

이후 pip install을 하면 시스템이 아니라 이 폴더 안에만 설치됩니다.
Kubeflow가 Jupyter 터미널을 제공하기 때문에 특별한 도구 없이 그냥 저 명령어만 치면 됩니다.

**Q. vLLM을 왜 안 쓰고 transformers로 바꿨나요?**
Kubeflow 환경에 gcc 컴파일러가 없어서 vLLM의 Triton JIT 컴파일이 실패했습니다.
sudo 권한도 없어서 설치가 불가능했고, 결국 transformers로 바꿔서 해결했습니다.
속도는 좀 느리지만 확실하게 동작합니다.

### 평가 관련

**Q. LLM-as-a-Judge가 공정한 평가 방법인가요?**
Teacher가 만든 데이터로 학습시켰으니 Teacher가 채점하면 편향이 있을 수 있다는 한계는 있습니다.
그래서 ROUGE-L, BERTScore 같은 객관적인 보조 지표를 같이 보는 겁니다.
완벽하진 않지만 도메인에 맞는 정성적 평가로는 가장 현실적인 방법입니다.

**Q. ROUGE-L이 뭔가요?**
생성 텍스트와 정답의 단어 겹침을 측정하는 지표입니다.
빠르고 계산이 쉽지만 의미보다 단어 겹침만 보는 한계가 있습니다.

**Q. Faithfulness가 뭔가요?**
모델이 지문에 있는 내용만 답변하는지 측정하는 지표입니다.
생성 답변의 단어 중 지문에 있는 단어 비율로 계산합니다.
이 값이 낮으면 지문 무시하고 내용을 지어냈다는 의미입니다. 할루시네이션 측정에 핵심 지표입니다.

**Q. 결과가 안 좋으면 어떻게 하나요?**
원인을 두 가지로 나눠서 봅니다.
데이터 품질 문제면 필터링 강화나 데이터 교체,
학습 설정 문제면 rank나 에폭, 학습률을 조정합니다.

### 기타

**Q. 실제로 AI Tutor에 붙이면 얼마나 좋아질 것 같나요?**
솔직히 아직 모릅니다. 결과가 나와봐야 알 것 같습니다.
기대는 할루시네이션이 줄고 단계적 설명이 가능해지는 것인데,
결과에 따라 추가 실험이 필요할 수도 있습니다.

**Q. Teacher로 GPT-4를 쓰면 더 좋지 않나요?**
맞습니다. 근데 OpenAI 약관상 API 출력으로 모델 학습 데이터를 만드는 게 제한됩니다.
그래서 오픈소스인 Qwen3-14B를 로컬에서 돌렸습니다.
비용도 없고 라이선스 문제도 없습니다.

**Q. 왜 Qwen3 모델을 선택했나요?**
세 가지 이유입니다.
한국어 성능이 오픈소스 중 상위권이고,
CoT thinking 모드를 공식 지원해서 추론 학습에 바로 쓸 수 있고,
Apache 2.0 라이선스라 제약이 없습니다.

### 모델 관련

**Q. 왜 하필 8B, 14B 조합인가요? 더 큰 Teacher를 쓰면 안 되나요?**
Teacher가 클수록 데이터 품질이 올라가는 건 맞습니다.
다만 14B도 L40S 48GB에서 fp16으로 충분히 돌아가고,
32B 이상은 메모리가 부족하거나 속도가 너무 느려집니다.
Student를 8B로 고른 이유는 로컬 환경에서 서빙이 가능한 현실적인 크기이기 때문입니다.

**Q. 8B 모델이면 충분히 좋은 성능이 나오나요?**
기대치를 어디에 두느냐에 따라 다릅니다.
저희 목표는 14B 수준과 완전히 동일한 성능이 아니라,
할루시네이션을 줄이고 단계적 설명이 가능한 수준입니다.
파인튜닝 전 베이스 8B 모델보다 나아지면 성공으로 보고 있습니다.

**Q. Qwen 말고 다른 모델은 왜 안 됐나요?**
LLaMA는 한국어가 약하고, EXAONE은 학교 서버에서 추론용으로 쓰고 있어서 파인튜닝 베이스로 쓰기 어렵습니다.
Gemma는 한국어 성능이 아쉽고, Qwen3가 한국어 + CoT + 라이선스 세 가지를 동시에 만족하는 유일한 선택지였습니다.

### 학습 과정 관련

**Q. 에폭이 뭔가요?**
전체 학습 데이터를 한 번 다 보는 단위입니다.
예를 들어 에폭 2라면 65,000개 데이터를 처음부터 끝까지 두 번 학습하는 겁니다.
많이 돌릴수록 더 많이 학습하지만 과적합 위험도 있습니다.

**Q. 에폭을 2번만 돌리는 게 충분한가요?**
LoRA 파인튜닝에서 데이터가 많으면 에폭이 적어도 충분한 경우가 많습니다.
너무 많이 돌리면 과적합 위험도 있습니다.
일단 2에폭으로 시작해서 결과가 부족하면 늘릴 계획입니다.

**Q. 학습 중에 중간 결과를 확인할 수 있나요?**
네, 에폭마다 자동으로 검증 세트에 대한 평가가 돌아가게 설정했습니다.
ROUGE-L, BERTScore, Faithfulness 지표가 에폭마다 기록됩니다.

**Q. GGUF 변환하면 성능이 떨어지지 않나요?**
양자화 방식에 따라 다르지만 저희가 쓰는 q4_k_m 방식은
원본 fp16 대비 품질 차이가 거의 없습니다.
용량은 절반 이하로 줄고 추론 속도는 빨라집니다.

### 개념 관련

**Q. CoT가 뭔가요?**
Chain-of-Thought, 단계적 추론이라는 뜻입니다.
"답은 A입니다" 대신 "왜냐하면 ~이고, 따라서 ~이므로, 결론적으로 A입니다"처럼
추론 과정을 포함해서 답변하는 방식입니다.
학생 입장에서는 과정을 보고 이해할 수 있어서 AI Tutor에 특히 적합합니다.

**Q. Ollama가 뭔가요?**
로컬에서 LLM을 서버로 띄워주는 도구입니다.
설치가 1분이고, GGUF 파일만 있으면 바로 REST API로 쓸 수 있습니다.
저희 AI Tutor가 내부적으로 Ollama API를 호출하는 구조입니다.

**Q. GGUF가 뭔가요?**
학습된 모델 가중치를 양자화해서 저장하는 포맷입니다.
원래 fp16 모델은 크기가 크고 고성능 GPU가 필요한데,
GGUF로 변환하면 크기가 줄고 일반 GPU나 CPU에서도 돌릴 수 있습니다.

**Q. Kubeflow가 뭔가요?**
학교 GPU 서버에 설치된 ML 워크플로우 플랫폼입니다.
GPU 자원 관리, 다중 실험 관리, 파이프라인 자동화를 지원합니다.
저희는 그 안에 Jupyter 노트북 서버를 띄워서 학습을 돌리고 있습니다.

### 개념 보충

**Q. fp16이 뭔가요?**
부동소수점 16비트 정밀도입니다.
숫자를 얼마나 정밀하게 표현하느냐의 단위인데,
fp32(32비트)가 더 정밀하지만 메모리를 두 배 씁니다.
fp16은 정밀도를 약간 낮추는 대신 메모리를 절반으로 줄일 수 있어서
GPU 메모리가 제한된 환경에서 학습할 때 표준적으로 씁니다.
실제 성능 차이는 거의 없습니다.

**Q. 배치 사이즈가 뭔가요?**
한 번에 몇 개의 데이터를 묶어서 학습할지 결정하는 값입니다.
배치 사이즈 4면 데이터 4개를 동시에 GPU에 올려서 한 번에 학습하는 겁니다.
클수록 학습이 빠르지만 메모리를 더 많이 씁니다.
저희는 배치 사이즈 2에 gradient accumulation 8을 써서
실질적으로 16개를 묶어서 학습하는 것과 같은 효과를 냈습니다.

**Q. 할루시네이션이 왜 생기나요?**
LLM이 학습할 때 "다음에 올 그럴듯한 단어"를 예측하도록 훈련되기 때문입니다.
지문을 보고 사실만 말하도록 제약이 없으면, 그럴듯하게 들리는 내용을 만들어냅니다.
파인튜닝으로 "지문에 없는 내용은 말하지 마라"를 직접 학습시키는 게 저희 목표입니다.

### 방향성 관련

**Q. 파인튜닝하면 무조건 원래 모델보다 좋아지나요?**
아닙니다. 데이터 품질이 나쁘거나 학습 설정이 잘못되면 오히려 나빠질 수 있습니다.
또한 특정 도메인에 특화될수록 다른 영역 성능이 떨어지는 trade-off도 있습니다.
그래서 파인튜닝 전후 비교 평가가 중요합니다.

**Q. DPO나 RLHF 같은 다른 방법은 왜 안 쓰나요?**
DPO나 RLHF는 SFT 이후에 추가로 하는 정렬 단계입니다.
좋은 답변과 나쁜 답변 쌍 데이터가 별도로 필요하고 구현도 복잡합니다.
시간이 촉박해서 일단 SFT 기반 파인튜닝에 집중하고 있습니다.

**Q. 4주 프로젝트인데 1주일도 안 남았는데 완성할 수 있나요?**
GPU 서버 대여가 6월 17일까지라 마감이 촉박합니다.
현재 데이터 생성이 진행 중이고, 완료되면 학습 2~3시간, 배포 30분이라 시간 안에 가능합니다.
다만 평가와 재학습까지 할 여유는 없을 수 있어서, 일단 학습 완료 후 기본 테스트까지만 목표로 잡고 있습니다.

**Q. 이후에 어떻게 발전시킬 계획인가요?**
이번 프로젝트는 단발성으로 진행하는 거라 추가 발전 계획은 없습니다.
학습이 완료되면 GGUF로 변환해서 개인 PC에 올려두고 결과를 확인하는 것으로 마무리할 예정입니다.

### 실용성 관련

**Q. 데이터 라이선스 문제는 없나요?**
사용한 데이터셋 모두 확인했습니다.
Wikipedia는 CC BY-SA, KorQuAD는 CC BY-ND, KLUE는 CC BY-SA, sae4K는 CC BY-SA, OpenThoughts는 Apache 2.0입니다.
Qwen3 모델도 Apache 2.0이라 비상업적 학습과 배포에 제약이 없습니다.

**Q. 학습 비용이 얼마나 드나요?**
학교 GPU 서버를 쓰기 때문에 추가 비용이 없습니다.
클라우드로 환산하면 A100 기준 시간당 약 3~4달러라
데이터 생성 13시간 + 학습 3시간 = 약 50~60달러 수준입니다.

**Q. 왜 영어 Wikipedia는 안 쓰고 한국어만 쓰나요?**
저희 AI Tutor가 한국어 강의 환경에서 쓰이기 때문입니다.
한국어 지문 기반으로 한국어로 답변하는 능력이 핵심이라 한국어 지문으로만 생성했습니다.
영어 데이터는 OpenThoughts로 추론 능력만 보강하는 방식으로 분리했습니다.
